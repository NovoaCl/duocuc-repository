¡Hola! 👋 Ese error ORA-01017: invalid username/password es muy típico en Oracle, y en tu caso (Docker + SQL Developer) casi seguro no es que la contraseña esté mal, sino cómo te estás conectando.

Te explico lo más probable 👇

🔴 Problema principal: estás conectando al contenedor equivocado (CDB vs PDB)

En tu init.sql haces esto:

ALTER SESSION SET CONTAINER = XEPDB1;

Eso significa que el usuario MDY2131_P3 se crea dentro del PDB XEPDB1, no en la base raíz (CDB).

👉 Entonces, si en SQL Developer te conectas sin especificar el PDB, Oracle intenta validar el usuario en el contenedor raíz → y ahí no existe, por eso el ORA-01017.

✅ Solución

Cuando crees la conexión en SQL Developer, usa:

Hostname: localhost
Port: 1521
Service name: XEPDB1 ← ⚠️ ESTO ES CLAVE
User: MDY2131_P3
Password: MDY2131.practica_p3
⚠️ Error común

Mucha gente pone:

Service name: XE ❌
o SID: XE ❌

Eso conecta al contenedor raíz → y tu usuario no existe ahí.

🧪 Cómo verificar que todo está bien en Docker

Puedes meterte al contenedor:

docker exec -it oracle-xe-p1 sqlplus system/oracle@XEPDB1

Y luego:

SELECT username FROM dba_users WHERE username = 'MDY2131_P3';

Si aparece → todo está OK 👍

🔎 Otra posible causa (menos probable)

Si sigue fallando, revisa:

1. ¿El script se ejecutó realmente?

A veces el init.sql no corre si el volumen ya existía.

👉 Solución:

docker-compose down -v
docker-compose up -d

Esto borra la data y vuelve a ejecutar el init.

2. Mayúsculas en usuario

Oracle guarda usuarios en mayúsculas por defecto:

MDY2131_P3  ✅
mdy2131_p3  ❌
3. Contraseña con comillas

La definiste así:

IDENTIFIED BY "MDY2131.practica_p3"

Eso hace la contraseña case-sensitive exacta.

💡 Resumen corto

Tu problema casi seguro es:

👉 Te estás conectando a XE en vez de XEPDB1.
